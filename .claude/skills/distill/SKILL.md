# Distill Conversation to Document

The user wants to extract and organize information from the current conversation into a document.

**Topic:** $ARGUMENTS

## Instructions

1. If `$ARGUMENTS` is empty, do NOT proceed. Instead, tell the user: "Please provide a topic, e.g. `/distill auth design for API`" and stop.

### Extract

2. Scan the entire conversation history and extract all information relevant to the topic.
   - **Privacy rule**: Never include personal user data — names, gender, age, passwords, API keys, tokens, email addresses, phone numbers, addresses, or any other private/sensitive information. Omit entirely.

### Organize

3. Organize the content in **logical order** (not chat order). Group related ideas together and build a coherent narrative or structure.
4. Choose the best document format for the content:
   - **Concept notes** — for learning and Q&A about a concept or technology
   - **Design doc** — for software architecture or system design discussions
   - **Decision log** — for discussions that led to specific decisions
   - **Tutorial/guide** — for how-to or step-by-step discussions
   - **Comparison/analysis** — for evaluating trade-offs between options
   - Or any other format that best fits the content.

### Compose

5. Add frontmatter with these four fields:
   ```yaml
   ---
   title: <auto-generated from topic>
   description: <one-sentence summary of the document>
   pubDatetime: <actual current UTC time — run `date -u +"%Y-%m-%dT%H:%M:%SZ"` to get it, never hardcode or approximate>
   tags: [<AI-generated relevant tags>]
   ---
   ```
6. Compose the document body in GitHub Flavored Markdown (GFM).
   - **GFM features**: Use tables, task lists, fenced code blocks (with language identifiers), and strikethrough when they make the document clearer. Don't force them where plain Markdown suffices.
   - **Diagrams**: Include a Mermaid diagram (in a `mermaid` fenced code block) when it helps illustrate structure, flow, or relationships.
   - **Emojis**: Use Unicode emoji characters (e.g., 🚀, ⚙️, ✅) inline where they aid scanning. Never use GitHub-style shortcodes (e.g., `:rocket:`) as they don't render outside GitHub.
   - **References**: Add link references for external URLs, heading anchors for cross-section links, or footnotes for supplementary notes where they help the reader navigate or find related context.

### Output

7. Auto-generate a concise, descriptive filename in kebab-case based on the topic (e.g., `auth-design.md`, `react-state-management.md`). Check for filename conflicts in the target directory and append a version suffix if needed (e.g., `auth-design-v2.md`).
8. Determine the output directory:
   - Check the `BLOG_DIR` environment variable. If it is not set in the shell, look for a `.env` file in the project root and read the `BLOG_DIR` value from it.
   - If `BLOG_DIR` is found (from either source): write to `<BLOG_DIR>/src/data/blog/<filename>.md`.
   - Otherwise: write to `docs/distilled/<filename>.md` (create directory if needed).
9. Tell the user the file path and a brief summary of what was captured.
