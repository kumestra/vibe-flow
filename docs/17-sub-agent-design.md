# Sub-Agent Design: The General Idea

## The One-Line Summary

A sub-agent is the agent loop called recursively — an LLM API call generated by AI, returning its result as a tool result.

---

## The Core Agent Loop (Recap)

```
while true:
    response = call_llm(messages, tools)
    if no tool calls → return response      ← done
    tool_results = execute_tools(response.tool_calls)
    messages += [response, tool_results]
```

A **tool** in this loop is normally a simple function:

```
input → compute → output
```

---

## What a Sub-Agent Actually Is

The "Agent tool" replaces that simple function with another full agent loop:

```
input (prompt) → [full LLM loop with its own tools] → output (final text)
```

In code, it's almost trivially simple:

```python
def agent_tool(prompt, tools):
    messages = [{"role": "user", "content": prompt}]
    while True:
        response = call_llm(messages, tools)
        if no tool calls:
            return response.text          # ← becomes tool_result for parent
        tool_results = execute_tools(response.tool_calls)
        messages += [response, tool_results]
```

From the **parent agent's perspective**, calling the Agent tool is identical to calling any other tool — it sends a `tool_use` block and gets back a `tool_result`. The parent never sees the sub-agent's internal loop.

---

## Who Talks to the Sub-Agent?

This is the key difference from the main agent loop:

| | Main agent | Sub-agent |
|---|---|---|
| Who provides user turns? | A human | The parent agent (via prompt) |
| Waits for input between turns? | Yes | No — runs to completion |
| Lifetime | Entire session | One task |

The sub-agent's conversation starts with a single message — the prompt the parent wrote — and runs autonomously until it returns a final answer. No human is in the loop.

```
# Main agent loop
user message → LLM → tools → LLM → ... → reply to user → wait → user message → ...

# Sub-agent loop
prompt → LLM → tools → LLM → ... → final answer (returned as tool_result)
```

---

## Why Use Sub-Agents?

Two concrete problems they solve:

### 1. Context Window Exhaustion

A long-running task fills the context window. Once it's full, the agent loses track of earlier work.

A sub-agent has its **own fresh context window**. The parent agent only sees the final summary — the sub-agent's entire internal conversation (potentially hundreds of messages) is discarded after it finishes.

### 2. Sequential Bottleneck

By default, tools run one at a time. If you need to research 5 things, you wait 5 turns.

Multiple sub-agents can run **in parallel**, each working independently:

```
Parent Agent
├── Sub-Agent A: analyze file X  ──┐
├── Sub-Agent B: analyze file Y  ──┤ all running at the same time
└── Sub-Agent C: analyze file Z  ──┘
         ↓ all finish
Parent synthesizes all three results
```

---

## The Contractor Analogy

Think of it like hiring a contractor:

- **You** (parent agent) describe the job in a brief
- **Contractor** (sub-agent) does the work with their own tools and resources
- **Contractor** hands you back the finished product
- You don't see *how* they did it — only the result

The sub-agent's system prompt can even differ from the parent's — like giving a contractor a task-specific scope of work rather than the full company handbook.

---

## The Design Variables

The core mechanic is simple. The complexity lives in how you configure each sub-agent:

| Variable | Question |
|---|---|
| **Tools** | Does it get all tools? A subset? Task-specific tools only? |
| **System prompt** | Same as parent? Stripped down? Task-focused? |
| **Parallelism** | Run one at a time or many simultaneously? |
| **Depth** | Can sub-agents spawn their own sub-agents? (Usually capped at 1–2 levels) |
| **Error handling** | What if the sub-agent fails or loops forever? |

## The Mental Model

```
Agent tool call (from parent's perspective):

tool_use  →  "Agent"  →  { prompt: "...", tools: [...] }
                                    ↓
                         [invisible sub-agent loop]
                                    ↓
tool_result  ←  { content: "here is the summary..." }
```

It's recursive agents, but the interface is flat. The parent sees one tool call and one result — the recursion is an implementation detail.

---

## What Makes It Elegant

The same loop that powers the main agent also powers sub-agents. There's no special "sub-agent runtime" — just the agent loop, called from within a tool. This means:

- Sub-agents get the full power of any tool (Read, Grep, Bash, etc.)
- Sub-agents can themselves call the Agent tool (nesting)
- The implementation is one recursive function, not a separate system

The insight: **agency is a property of the loop, not of the entity running it**.
